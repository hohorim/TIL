# <SERVICE_DOMAIN> 504 (Gateway Timeout) 장애 대응 기록

- **일자**: 2026-07-21
- **영향 서비스**: <SERVICE_DOMAIN> (bim / <FRONTEND_SVC>)
- **증상**: `GET /` 요청 시 nginx ingress에서 `upstream timed out (110: Operation timed out) while connecting to upstream` 발생, 클라이언트에는 504 응답

---

## 1. 증상

```
2026/07/21 00:59:28 [error] upstream timed out (110: Operation timed out) while connecting to upstream,
client: <NODE-C_IP>, server: <SERVICE_DOMAIN>, upstream: "http://<POD_IP>:80/", host: "<SERVICE_DOMAIN>"
...
<NODE-C_IP> - - "GET / HTTP/1.1" 504 562 ... 5.000, 5.001, 5.001  504, 504, 504
```

- 응답 지연이 아니라 **TCP 연결 자체가 성립되지 않는** 상태 (5초 × 3회 재시도 후 15초 만에 504)
- ingress-nginx 파드 내부에서 직접 curl 재현 확인:
  ```
  curl -v --connect-timeout 3 http://<POD_IP>:80
  * connect to <POD_IP> port 80 failed: Operation timed out
  ```

---

## 2. 원인 분석 과정

### 2-1. 대상 파드 확인
- 504의 upstream IP(`<POD_IP>`)는 `bim` 네임스페이스의 `<FRONTEND_POD>` 파드
- `<NODE-D>` 노드에서 `1/1 Running`, `Ready: True` — 파드 자체는 정상

### 2-2. Calico 상태 확인
- `<NODE-D>`의 `<CALICO_POD_OLD>` 파드가 **`0/1 Not Ready`** (39일간 지속)
- Readiness probe 실패 로그:
  ```
  Number of node(s) with BGP peering established = 0
  BGP not established with <NODE-A_IP>, <NODE-C_IP>, <NODE-B_IP>, <NODE-E_IP>
  ```
  → `<NODE-D>`이 클러스터의 나머지 4개 노드 전부와 BGP 피어링이 안 되어 있는 상태였음

### 2-3. 근본 원인 특정
- 노드 annotation과 실제 노드 IP 비교:

| 노드 | annotation (`projectcalico.org/IPv4Address`) | 실제 IP | 일치 여부 |
|---|---|---|---|
| <NODE-A> | <NODE-A_IP>/24 | <NODE-A_IP> | ✅ |
| <NODE-B> | <NODE-B_IP>/24 | <NODE-B_IP> | ✅ |
| <NODE-C> | <NODE-C_IP>/24 | <NODE-C_IP> | ✅ |
| **<NODE-D>** | **<WRONG_IP>/24** | **<NODE-D_IP>** | ❌ **불일치** |
| <NODE-E> | <NODE-E_IP>/24 | <NODE-E_IP> | ✅ |

- `<NODE-D>`만 Calico가 BGP 라우터 ID로 광고하는 IP(annotation)가 실제 노드 IP와 어긋나 있었음
- 이로 인해 다른 노드들이 `<NODE-D>`과 BGP 세션을 맺지 못했고, 결과적으로 `<NODE-D>` 위 파드 서브넷(<POD_SUBNET>)에 대한 라우팅이 다른 노드에 전파되지 않아 **connect timeout → 504**로 이어짐
- 다른 4개 노드의 calico-node는 모두 `1/1 Running`, annotation 일치 — 클러스터 전역 문제가 아닌 `<NODE-D>` 단일 노드 문제로 확정

### 2-4. 사전 안전 점검
조치 전 리스크 확인:
- `<NODE-D>` 리소스: CPU 54%, Memory 47% — 여유 있음
- `MemoryPressure` / `DiskPressure` / `PIDPressure` 전부 `False`
- 해당 노드에 istiod, rancher, prometheus, bim-service 등 다수 파드가 함께 떠 있어 calico-node 재시작 시 순간적인 네트워크 흔들림의 영향 범위가 넓다는 점을 인지하고 진행

---

## 3. 해결 방법

```bash
# 1. annotation을 실제 노드 IP로 수정
kubectl annotate node <NODE-D> projectcalico.org/IPv4Address=<NODE-D_IP>/24 --overwrite

# 2. calico-node 파드 재시작 (annotation 재적용 강제)
kubectl delete pod -n kube-system <CALICO_POD_OLD>
```

- 재시작 직후에는 BIRD가 세션을 다시 맺는 데 약 2분 정도 소요되며, 그 사이 readiness probe 실패가 몇 차례 발생하는 것은 정상 과정이었음 (마지막에 정상 확립되며 `Ready: False → True`로 전환)

---

## 4. 결과 확인

- `calico-node` 파드 로그에서 4개 노드 전체와 BGP mesh 세션 정상 확립 확인:
  ```
  Mesh_<NODE-A_IP>: State changed to up
  Mesh_<NODE-B_IP>: State changed to up
  Mesh_<NODE-C_IP>: State changed to up
  Mesh_<NODE-E_IP>: State changed to up
  ```
- annotation 유지 확인: `<NODE-D_IP>/24`
- Pod 상태: `<CALICO_POD_NEW>` → `1/1 Running` (Ready)
- ingress-nginx → `<POD_IP>:80` curl 재검증 → 정상 응답 확인, 504 재현 안 됨

---

## 5. 참고 / 향후 조치 아이디어

- 이 노드는 과거(`<NODE-D>`)에도 동일한 Calico annotation 불일치로 인한 장애 이력이 있었음 (Redis `Connection reset`, 광범위 504 등). **재발성 이슈**이므로 근본 원인(왜 이 노드에서만 annotation이 틀어지는지 — 인터페이스 자동탐지, 노드 재부팅/IP 변경 이력 등) 파악이 필요.
- IP 자동탐지 방식을 명시적으로 고정하는 것을 고려:
  ```bash
  kubectl set env daemonset/calico-node -n kube-system IP_AUTODETECTION_METHOD=cidr=<NODE_SUBNET>
  ```
- Calico 노드 BGP peering 상태에 대한 알림(alert)을 모니터링에 추가하면, 39일간 방치되었던 이번 케이스처럼 조용히 지속되는 라우팅 장애를 조기에 감지할 수 있음.
