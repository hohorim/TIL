@ 기본개념
- Dababase: 컬렉션의 물리적 컨테이너.
	하나의 데이터베이스에 보통 여러개 컬렉션 존재

- Collection: Document의 그룹
	Table과 같은 개념
	스키마를 딱 하나만 지정하는 듯한 강요는 없어서, 컬렉션 내부 도큐먼트들의 스키마는 다를 수 있고, 서로 다른 필드를 가짐.
	* 스키마: 해당 컬렉션의 문서에 어떤 종류의 값이 들어가는지를 정의

- View: 다른 컬렉션 또는 뷰의 집계 파이프라인에 의해 콘텐츠가 정의되는 읽기 전용 쿼리 개체

- Document:	하나의 key와 값의 집합.
	동적 스키마(동일한 컬렉션 내의 도큐먼트가 동일한 필드 또는 구조를 가질 필요없음, 동일한 필드안에 다른타입의 데이터 보유가능)
	Tuple/Row 와 같은 개념
	
- Field:
	Column 과 같은 개념
	
- Bucket: 일반적으로 파일 저장소는 bucket 이라고 지칭
	GridFs 도 bucket 이라고 지칭
	bucket 은 일종의 컬렉션 중 하나로 파일의 청크를 담고 있는 저장소입니다.
	즉 fs.files 와 fs.chunks 는 하나의 bucket 으로 봄.

- 함께 조회되는 경우 빈번한 데이터들은 같은 collection 에 담아. query 횟수를 줄일 수 있다.
- 주로 읽기만 하는 데이터와 자주 업데이트하는 데이터는 별개의 collection에 담는다.

- Entity 간의 relation 을 collection 간에 reference 할지, embed 할지 결정
	- reference: colletion 간 참조할 수 있도록 id 저장 (쓰기에 빠름, document 하나만 수정/query 여러번 실행)
	- embed: 관계된 document를 통째로 저장 (읽기에 빠름, query 한번에 모두조회/document 중복 수정작업 필요)
	- 잦은 수정이 일어난다면 reference 가 더 바람직
	
- 한 Document 의 최대 저장할 수 있는 Size 는 16MB(BSon 문서 제한 크기임)
- database 최대 64 TB

-BSON: 
	배열 및 중첩 Document 와 같은 복잡한 데이터 유형을 효율적으로 저장할 수 있는 유연한 JSON 과 유사한 형식인 BSON(binary json)으로 저장.
	?? json이 아닌 bson 사용 이유
	1. json 은 제한된 수의 기본 데이터 유형만 지원. 특시 날짜 및 이진 데이터에 대한 지원 부족
	2. json 개체 및 속성에는 고정 길이가 없으므로 순회 속도가 느려짐.
	
	json은 사람, 기계가 읽을 수 있지만, bson은 기계만 읽기 가능.
	json 보다 bson 이 데이터 유형지원이 훨씬 많이 됨.
	

- MongoTemplate vs MongoRepository

- primary Key
	mongoDB 는 _id 라는 필드를 같이 insert 해주지 않으면 mongodb 클라이언트에서 ObjectId 타입으로 자동으로 만들어진다.
	unix 날짜정보, 클라이언트 고유 ip/mac 번호, processid, 자동증가 값으로 구성.
	Entity Annotaion으로 @Id 라고 지칭할 수 있는데, 만약 자동이 아닌 직접 _id 값을 넣는다면
	@MongoId(value = FieldType.STRING) 로 선언해야함.
	그렇지 않으면, 타입이 달라져서 find 해오지 못하고 계속 null 값이 return 됨.
	
	
- JAVA로 MongoDB에서 데이터 조회할 때, 자식 클래스 중 어떤 클래스와 매핑할 지 결정하는 것은  Document 내 저장되어 있는 _class이다. 
	_class는 JAVA에서 데이터 저장 시, Document에 자동으로 같이 저장된다.
	_class로 구분하기 때문에, 자식 클래스의 클래스명은 변경하면 안된다. 변경 시, 기존 저장된 데이터를 매핑해서 가지고오지 못한다.

- 인덱스는 읽기이벤트가 많이 일어나는 collection에 유리함.
	반면에, 쓰기 이벤트가 많이 일어나는 collection에 설정하게 되면, update 혹은 insert 될 때마다 
	index가 재생성되어야 함으로 비용이 많이들음.
	
- BulkOps: 다량의 데이터를 한번에 업데이트/삭제 시 사용. I/O 작업 줄이고, 데이터 처리속도 향상
	BulkOperations bulk = mongoTemplate.bulkOps(BulkOperations.BulkMode.UNORDERED, OwnerEntity.class); 
		-> UNORDERED: 작업을 순서대로 하지않고, 가능한 한 많은 작업을 동시실행
