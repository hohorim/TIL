- Entity: 개별성, 변화가능성
- 표현 -> 응용 -> 도메인 -> 인프라스트럭처
	
- 애그리거트는 작게 만들고, 최대한 나누는 것이 좋다고 한다.
- 인프라에 대해서는 루트 엔터티 말고는 따로 관리할 수가 없다
- Repository: 무조건 루트 엔터티를 거쳐서 내부 엔터티들이 수정되어야 하는 구조다.
- domain 이 infra 를 알면 안된다.

- domain.repository; repository 인터페이스
			  infra; repository 구현체
			  
- 고수준모듈(응용, 도메인), 저수준모듈(인프라스트럭처)
- repsotiroy interface가 호출되는 곳은 고수준이어야한다.

- 표현계층, 응용계층에 도메인 로직을 넣지 말라.
- 응용계층은 표현 계층에 필요한 데이터만 전달한다.
- 보통 표현계층은 값 검증
- 응용 계층은 논리적 검증/존재 검증

- 도메인 계층에는 도메인 모델과 도메인 서비스가 존재
- 저수준 추상화하는 것 경계-> 그럼 도메인이 구현체에 의존하게됨.

- 핵심 서비스 로직에서 다른 애그리거트 핵심 로직에 있는 메서드를 중복 구현하지말자
	- 필요하다면 다른 애그리거트의 도메인 서비스를 생성하여 기능 구현
	
	
- 팩토리 메소드 패턴에서는 객체를 생성하기 위한 인터페이스를 정의하는데, 어떤 클래스의 인스턴스를 만들지는 서브 클래스에서 결정하게 만드는 패턴이다.


- 팩토리에 command 는 가지말고
- 모든 엔터티 변경은 팩토리에서
- 한 트랜잭션에서 한 애그리거트 내부에서 다른 애그리거트 상태를 변경하는 기능을 하면 안됨.
	-> 굳이 그래야한다면 응용서비스에서 애그리고트를 수정하도록 구현. 서비스를 하나 더파란 얘기.
	
- 응용서비스 기능과 도메인 기능은 분리하자.
	응용기능은 생성/수정에 대한 db 메서드(생성기능, repository 메서드) 호출 등이고,
	도메인 기능은 생성가능한지(중복키 여부)의 여부 등이다.
	
	
- 도메인서비스: 한 애그리거트에 넣기 애매한 도메인 기능을 억지로 구현하면안되서, 그에대한 해결책
	- 상태 없이 로직만 구현. 상태는 다른 방법으로 전달받는다.
	- 위치: domain.service(자식 주체) 

- approveflow: 결재 플로우, 결재 기본 담당자 
- approveitem: 결재 아이템, 결재 담당자, 결재 참조자, 결재 이슈, 결재 이력
- container: 성과품, 성과품 이력, 연관성과품, 성과품메타데이터, 성과품마일스톤, 공유성과품
- file: 파일, 변환파일, 공유파일
- folder: 폴더, 폴더권한
- issue: 이슈, 이슈 담당자, 이슈 상태, 이슈 댓글, 이슈 이미지, 이슈 연관참조
- milestone: 마일스톤, 마일스톤 태스크, 마일스톤메모
- project: 프로젝트, 프로젝트 담당자
- team: 팀, 팀 유저, 팀폴더
- user: 사용자, 유저 폴더
- work: 업무, 업무 아이템, 업무 댓글


- projectcode: 프로젝트 코드
- projectcontainer: 프로젝트 컨테이너 코드, 프로젝트 컨테이너 아이템 코드
- projectmetadata: 프로젝트 메타데이터 코드
- projectchecklist: 프로젝트 체크리스트 아이템 코드

- poolwork: 업무 타입, 업무 카테고리
- poolsystem: 시스템 그룹코드, 시스템 코드
- poolmetadata: 메타데이터 pool 코드
- poolcontainer: 컨테이너 pool 코드, 컨테이너 pool 아이템 코드
- poolchecklist: 체크리스트 pool 타입코드, 체크리스트 pool 아이템 코드
